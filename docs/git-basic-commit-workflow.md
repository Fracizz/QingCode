# Git 基础提交与推送流程

> 状态：**已实现**（2026-07-22）。
> 决策：支持暂存、取消暂存、本地提交和向已配置 upstream 推送；提交时只提交已暂存内容。

## 1. 背景

QingCode 当前的 Git 集成以只读查看为主：

- 源代码管理面板展示工作区变更
- 资源管理器、编辑器标签和状态栏展示 Git 状态
- 支持查看 HEAD 与工作区之间的差异
- 不支持暂存、取消暂存或提交

本次增强补齐最小可用的提交闭环，并允许复用系统 Git 已有的远程/upstream 配置执行推送，同时不引入远程凭据管理、冲突处理和高风险恢复操作。

## 2. 产品目标

用户可以在源代码管理面板中完成：

1. 区分已暂存与未暂存的更改
2. 暂存单个文件或全部文件
3. 取消暂存单个文件或全部文件
4. 输入提交信息
5. 仅提交已暂存内容
6. 独立推送当前分支，或在提交成功后继续推送
7. 在操作完成后立即看到最新 Git 状态

## 3. 范围

### 3.1 本次实现

| 功能 | 要求 |
|---|---|
| 状态分组 | 显示“已暂存的更改”和“更改”两组 |
| 单文件暂存 | 文件行按钮和右键菜单均可执行 |
| 全部暂存 | “更改”分组提供全部暂存操作 |
| 单文件取消暂存 | 文件行按钮和右键菜单均可执行 |
| 全部取消暂存 | “已暂存的更改”分组提供全部取消暂存操作 |
| 丢弃更改 | 右键菜单「丢弃更改」；执行前二次确认；未跟踪文件删除，已跟踪文件恢复 |
| 面板布局 | “更改”和“已暂存的更改”上下分区、独立滚动并可折叠，提交区固定在底部 |
| 提交信息 | 面板底部提供多行输入框 |
| 本地提交 | 只提交暂存区，不自动暂存未暂存文件 |
| 推送 | 工具栏可独立推送；提交区可选择提交成功后推送，均使用当前分支已配置 upstream |
| 状态刷新 | 每次 Git 写操作后执行完整状态刷新 |
| 错误反馈 | 展示 Git 原始错误的可读摘要，并保留用户输入 |

### 3.2 明确不做

- pull、fetch 和远程凭据管理
- 分支创建、切换、合并和 rebase
- stash
- amend、签名提交和交互式暂存
- 冲突解决器

推送只调用普通 `git push`，不猜测远程仓库、不自动设置 upstream，也不接管 Git 的凭据流程。其余功能涉及认证、冲突、历史修改，不属于本次简单增强。丢弃更改属于有二次确认的显式数据丢失操作，已纳入实现范围。

## 4. 交互方案

源代码管理面板从上到下排列：

```text
源代码管理  当前分支           推送 刷新
更改          3                 [+]  ← 可折叠
  M  src/a.ts
  D  src/c.ts
  U  src/new.ts

已暂存的更改  2                 [-]  ← 可折叠
  M  src/a.ts
  A  src/b.ts

☑ 提交后推送
[ 提交信息输入框                     ]
[ 提交并推送到 当前分支              ]
```

### 4.1 提交规则

- 提交信息去除首尾空白后不能为空。
- 暂存区为空时不可提交，并提示“请先暂存要提交的更改”。
- 点击提交不会自动暂存任何文件。
- 提交成功后清空提交信息并刷新状态。
- 提交失败时保留提交信息与当前列表，展示错误原因。
- Git hooks 正常执行；hook 失败视为提交失败，并显示 hook 输出。
- “提交后推送”默认开启；提交成功后才执行推送。推送失败不会撤销已经成功的本地提交，并明确显示“提交成功，但推送失败”。

### 4.2 推送规则

- 工具栏推送按钮和“提交后推送”都执行不带远程/分支参数的 `git push`。
- Git 根据当前分支已有 upstream 和用户凭据配置决定目标；未配置 upstream 或认证失败时原样展示可读错误。
- 推送不执行 `--force`、不自动设置 upstream，也不修改工作区或暂存区。

### 4.3 文件操作

- “更改”组文件行提供暂存按钮。
- “已暂存的更改”组文件行提供取消暂存按钮。
- 同一操作同步加入文件右键菜单，保证按钮不明显时仍可发现。
- 分组标题提供全部暂存或全部取消暂存。
- 操作进行时禁用重复点击，并显示轻量加载状态。
- 文件点击仍沿用现有差异查看行为。

### 4.4 双状态文件

Git 的 porcelain 状态包含索引区和工作区两列 `XY`。例如：

- `M `：文件已暂存，工作区没有后续修改
- ` M`：只有未暂存修改
- `MM`：已暂存后又继续修改
- `AM`：新增文件已暂存后又继续修改
- `??`：未跟踪文件

`MM`、`AM` 等文件必须同时出现在两组中：

- 已暂存组代表 `HEAD → index`
- 更改组代表 `index → worktree`

对其中一组执行操作后，另一组仍按 Git 实际状态保留。

## 5. 状态模型调整

当前 `git.rs` 的 `parse_status` 会对 `XY` 调用 `trim()`，`git_status.rs` 也会把部分双列状态压缩成单字符。这会导致 `M ` 与 ` M` 都变成 `M`，无法判断更改位于暂存区还是工作区。

实现前必须统一状态契约：

- Rust 返回完整的两字符 `XY`，特殊状态保留 `??` 和 `!!`
- 前端以 `indexStatus` 和 `worktreeStatus` 辅助函数解释 `XY`
- 现有树、标签和角标继续通过统一 glyph 函数显示单个用户友好字母
- SCM 分组只依据完整 `XY`，不依据压缩后的展示字符
- 轻量状态接口和完整状态接口必须返回一致的状态语义

建议继续保留 `GitChange.status: string`，避免扩大序列化结构；但该字段的新契约必须明确为未经裁剪的 porcelain `XY`。

### 6.1 新增 Tauri 命令

#### `git_stage`

参数：

```text
path: 项目根目录
files: 项目内相对路径数组
```

行为：

- 校验项目根目录位于 `PathAllowlist`
- 校验每个文件路径不为空、无父目录穿越且位于项目内
- 执行 `git add -- <files...>`
- 全部暂存由前端传入当前未暂存文件列表，不使用隐式全仓库 shell 命令

#### `git_unstage`

参数与 `git_stage` 相同。

行为：

- 优先执行 `git restore --staged -- <files...>`
- 对尚无 HEAD 的首次提交仓库使用兼容回退
- 不修改工作区文件内容

#### `git_discard`

参数：

```text
path: 项目根目录
files: 项目内相对路径数组
staged: 是否丢弃已暂存侧（true）或工作区侧（false）
```

行为：

- 校验项目根与路径同 `git_stage`
- 存在未合并冲突的路径时拒绝
- `staged=false`：对未跟踪文件删除磁盘内容；对已跟踪工作区更改执行 `git restore -- <files...>`
- `staged=true`：有 HEAD 时执行 `git restore --source=HEAD --staged --worktree -- <files...>`；无 HEAD 时取消暂存并删除工作区文件
- 前端在执行前通过 `ConfirmDialog` 二次确认

#### `git_commit`

参数：

```text
path: 项目根目录
message: 提交信息
```

行为：

- 后端再次校验提交信息非空
- 通过进程 stdin 向 `git commit --file=-` 传递提交信息
- 不拼接 shell 字符串，避免命令注入和转义问题
- 不添加 `-a`，确保只提交暂存区
- 保留 Git hooks、用户身份检查和仓库配置行为
- 返回提交成功摘要；失败时返回 stderr/stdout 中可读的错误信息

#### `git_push`

参数：项目根目录 `path`。

行为：

- 校验项目根目录位于 `PathAllowlist`
- 执行普通 `git push`，使用当前分支已配置的 upstream
- 不通过 shell，不自动添加远程、分支、`--set-upstream` 或 `--force`
- 返回推送摘要；失败时返回 stderr/stdout 中可读的错误信息

### 6.2 命令执行约束

- 继续使用系统 Git CLI，不引入 libgit2
- Windows 保持 `CREATE_NO_WINDOW`
- 所有文件参数必须位于 `--` 之后
- Git 写操作放入 `spawn_blocking`，避免阻塞 Tauri 异步运行时
- 不通过 shell 执行命令
- 不静默吞掉写操作错误

### 6.3 命令注册

- `git::git_stage`
- `git::git_unstage`
- `git::git_commit`
- `git::git_push`

### 7.1 状态辅助函数

- 判断文件是否存在未暂存更改

这些函数必须覆盖 `M `、` M`、`MM`、`AM`、`AD`、`R `、`??` 等状态。

### 7.2 面板状态

`SourceControlPanel` 增加：

- `commitMessage`
- 当前 Git 操作类型
- 当前正在处理的文件
- 分组折叠状态和“提交后推送”选项
- 提交、推送、暂存和取消暂存处理函数

Git 写操作统一走：

```text
校验输入
  → safeInvoke
  → 完整 git_status 刷新
  → 同步 gitStatusStore/sourceControlStore
  → 成功或失败反馈
```

写操作后必须使用完整 `git_status`，避免轻量缓存短暂丢失 staged / unstaged 分组信息。

### 7.3 列表结构

现有单个虚拟列表改为两个可独立为空、独立滚动和折叠的分组：

- 已暂存的更改
- 更改

每组使用固定行高虚拟列表，分组标题保持可见；两组展开时平分可用高度，一组折叠时另一组获得剩余空间。提交区固定在底部，不随文件列表滚动。

### 7.4 反馈方式

- 成功：使用现有 toast，例如“已暂存 3 个文件”“提交成功”
- 输入或状态问题：在提交区就地提示
- Git 命令失败：使用 toast，并在面板错误区域保留可读信息
- 不使用浏览器原生 `alert`、`prompt` 或 `title`
- 悬停说明继续使用共享 `Tooltip`

## 8. 安全与异常处理

| 场景 | 预期行为 |
|---|---|
| 项目不是 Git 仓库 | 不展示提交操作，沿用当前空态 |
| 系统未安装 Git | 展示“无法运行 Git”错误，不清空输入 |
| Git 用户名或邮箱未配置 | 展示 Git 的身份配置错误 |
| pre-commit hook 失败 | 提交失败，显示 hook 输出并保留消息 |
| 暂存文件随后被删除 | 以 Git 命令结果为准，失败后刷新状态 |
| 文件名包含空格或中文 | 通过参数数组和 `--` 正常处理 |
| 文件名以 `-` 开头 | `--` 阻止其被解释为 Git 参数 |
| 路径越过项目根目录 | 后端拒绝操作 |
| 仓库尚无首次提交 | 支持暂存、取消暂存和首次提交 |
| 提交期间再次点击 | 按钮禁用，不启动并发提交 |
| 当前分支未配置 upstream | 推送失败，保留本地提交并展示 Git 提示 |
| 远程认证或 hook 拒绝操作 | 展示 Git 输出，不吞掉错误 |

## 10. 验收标准

- 用户无需离开 QingCode 即可完成一次本地提交
- 用户可将当前分支推送到已经配置的 upstream，且不会隐式创建或改写远程配置
- 提交绝不会隐式包含未暂存文件
- `MM`、`AM` 等双状态文件展示和操作正确
- 取消暂存不会丢失工作区内容
- 所有写操作均受项目路径白名单保护
- Git hooks 和身份配置错误可见且可理解
- 写操作完成后面板、资源管理器、标签和状态栏状态一致
- 现有 Git diff、文件打开、右键菜单和自动刷新功能无回归
