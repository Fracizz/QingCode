# QingCode 发布验收清单

该清单用于准备下一个版本，不创建标签、不发布到远程。使用临时项目和临时 Git 仓库，避免测试写入真实工作目录。

## 1. 发布前检查

1. 分离与本次发布无关的工作区改动，特别是临时 `.pdb`、本机脚本和未完成的配置。
2. 如需发版，先使用 `pnpm bump:version -- <version>` 保持 `package.json`、`src-tauri/Cargo.toml` 与 `src-tauri/tauri.conf.json` 一致；未确认发版前不修改版本号。
3. 运行：

   ```powershell
   pnpm check
   pnpm build
   pnpm package:exe
   pnpm smoke:start -ExePath release/QingCode.exe -WaitSeconds 4
   # 多架构正式包由 GitHub Actions 构建：Windows x64 / Windows ARM64 / macOS arm64
   ```

## 2. 桌面应用手工验收

| 场景 | 预期结果 |
| --- | --- |
| UTF-8、UTF-8 BOM、GBK 中文、含 GB18030 独有字符的文件 | `auto` 能打开文本并显示实际编码；手动「重新打开」不写盘，「转换」只在保存时生效 |
| 编辑中的文件被外部工具改写 | 干净标签自动重载；脏标签可选比较、重载或保留本地修改 |
| 含中文名、空格名、重命名、未跟踪、已删除、已暂存修改的 Git 仓库 | 源代码管理清单、差异和 HEAD 并排比较对应正确文件 |
| 未信任项目 | 只能浏览；编辑、终端和运行脚本受限；信任后恢复使用 |
| 多项目切换后重启应用 | 各项目的标签、光标位置和终端元数据能恢复 |

## 3. 发布判定

- 所有自动检查通过，无未分析的工作区改动。
- 上表手工场景均通过，记录测试版本号、Windows 版本与 WebView2 情况。
- 完成版本号更新后，再创建对应 `v<version>` 标签并触发 Release 工作流。
